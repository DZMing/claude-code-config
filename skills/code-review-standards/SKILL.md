# 分层代码审查标准

## 审查优先级

### CRITICAL（阻止提交）

- **硬编码凭证**（api_key/secret/password）→ 用环境变量
- **数据丢失操作**（rm -rf/DROP TABLE/DELETE FROM）→ 加确认+备份
- **无限循环**（while(true) 无退出条件）→ 加退出条件/超时

### HIGH（应该修复）

- 函数 >50 行 → 拆分为更小单元
- 文件 >800 行 → 模块化
- 嵌套 >4 层 → 早返回/提取函数
- 缺少错误处理（fetch/JSON.parse 无 try-catch）→ 包裹外部调用
- 代码重复 >=3 次 → 提取公共函数

### MEDIUM（建议改进）

- 缺少类型注解 → 添加 TypeScript 类型或 JSDoc
- 魔法数字 → 命名常量 `MAX_RETRY_COUNT`
- O(n^2) 可优化 → 用 Set/Map 降到 O(n)
- 缺少边界测试 → 补充空值/极值/特殊字符用例
- 缺少缓存 → 重复查询/计算加缓存

### LOW（可选）

- 命名不够描述性（data1, temp2）→ 用业务语义命名
- 缺少注释（复杂逻辑无 Why 注释）
- 行过长 >120 字符 → 拆分或提取变量
- 尾随空格 → 格式化工具自动修复

## 质量门禁

| 指标 | 阈值 | 行为 |
|------|------|------|
| CRITICAL 问题数 | >0 | **阻止提交** |
| 测试覆盖率 | <80% | **阻止提交** |
| HIGH 问题数 | >=3 | 需要修复后提交 |
| 函数行数 | >50 行 | 警告 |
| 文件行数 | >800 行 | 警告 |

## 审查流程

```
代码变更
  ↓
1. CRITICAL 检查 → 发现 → 立即阻止，必须修复
  ↓ 无问题
2. HIGH 检查 → 发现 → 警告，询问是否修复
  ↓
3. MEDIUM 检查 → 发现 → 提供建议，用户选择
  ↓
4. LOW 检查 → 发现 → 记录日志，不阻塞
  ↓
通过 → 允许提交
```

## 安全检查模式

检测硬编码凭证的关键模式：
- `api_key = "..."` / `apiKey: "..."`
- `secret = "..."` / `password = "..."`
- `token = "sk_live_..."` / `token = "pk_test_..."`

检测到后必须：
1. 移除硬编码值
2. 改为 `process.env.API_KEY` 或 `.env` 文件
3. 确认 `.env` 在 `.gitignore` 中

## 批量审查（commit/PR 时）

通过标准：
- CRITICAL 问题：0 个
- HIGH 问题：< 3 个
- 测试覆盖率：>= 80%
- 代码质量评分：>= 7/10
