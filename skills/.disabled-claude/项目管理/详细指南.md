## 📊 项目管理标准化系统（AI 项目经理模式）

### 为什么需要标准化？

**核心痛点：AI 失忆**

你有没有遇到这些问题：

**❌ AI 每次从零开始**
- 用户: "继续开发用户管理功能"
- AI: "好的，我来实现用户管理的实体类..."
- 用户: "等等！实体类我昨天就写完了！"
- AI: "抱歉，我不知道你已经完成了哪些部分..."

**❌ 不知道项目有哪些模块**
- 用户: "帮我看看还有哪些功能没做完"
- AI: "我需要扫描整个项目...（扫描 5 分钟）...发现了这些..."
- 用户: "不对，这些是测试代码，核心业务在 src/ 目录！"
- AI: "抱歉，我没有项目结构的上下文..."

**❌ 进度评估全靠感觉**
- 用户: "项目完成度多少了？"
- AI: "扫描代码...大概 60%？"
- 用户: "能给个准确数字吗？"
- AI: "..."

**❌ 文档更新麻烦**
- 用户开发完功能，提交代码
- 第二天: "昨天做了什么来着？"
- 翻 Git 记录，手动整理文档 30 分钟
- 文档和代码经常不同步

**💡 核心问题：AI 再强大，也需要上下文！**

**关键认知：**
✅ AI 可以帮你写代码、调试错误
❌ 但 AI 不知道项目做到哪了、有哪些模块、昨天做了什么

**解决方案：**
给 AI 提供**标准化的项目管理上下文**，让 AI 成为你的**项目经理**，而不仅仅是代码助手。

### 三大标准文档 📚

每个项目都应该有的三个文档（放在 `docs/` 目录）：

#### 1. 需求文档 - 明确"要做什么"

**作用：**
- 团队对需求达成一致
- AI 可以关联代码实现和需求
- 需求变更有迹可循

**核心字段：**
```markdown
## 1. 功能需求

- [ ] REQ-001: 用户管理模块
  - 优先级: 高
  - 预计时间: 5天
  - 状态: 待开发
  - 描述: 实现用户的增删改查
  - 验收标准:
    - [ ] 支持用户增删改查
    - [ ] 支持角色分配
    - [ ] 接口响应时间 < 500ms
```

#### 2. 项目状态 - 记录"做到哪了" ⭐⭐⭐⭐⭐

**关键特性：**
⭐ AI 自动更新：开发完运行 `/update-status`，AI 自动扫描代码更新文档
⭐ 精确进度：按模块分类统计，计算完成百分比
⭐ 完整记录：每个任务包含完成时间、耗时、说明

**文档结构：**
```markdown
# 项目状态

最后更新: 2025-01-14 15:30

## 📊 当前状态
- 项目阶段: 开发中
- 整体进度: 72%
- 下一步计划: 完成用户反馈前端页面

## ✅ 已完成
- [x] 用户管理模块 (2025-01-10)
  - 耗时: 2天
  - 说明: 完成 Entity/Service/Controller，支持增删改查

## 🚧 进行中
- [ ] 用户反馈模块 (开始: 2025-01-12)
  - 进度: 70%
  - 预计完成: 2025-01-15
  - 当前工作: 正在实现 Service 层

## 📋 待办
- [ ] 添加单元测试
  - 优先级: 中
  - 预计工作量: 2天
```

#### 3. 待办清单 - 规划"接下来做什么"

**文档结构：**
```markdown
# 待办事项清单

## 🔥 高优先级（紧急重要）
- [ ] 修复生产环境登录异常
  - 截止日期: 2025-01-15
  - 描述: 用户反馈无法登录系统

## 📌 中优先级（重要不紧急）
- [ ] 实现用户权限管理功能
  - 截止日期: 2025-01-20

## 💡 低优先级（可延后）
- [ ] 优化首页加载速度
  - 截止日期: 2025-02-01

## 🔄 进行中
- [ ] 开发数据统计模块（进度: 60%）
  - 开始日期: 2025-01-10
  - 预计完成: 2025-01-18

## ✅ 最近完成
- [x] 实现用户登录功能 (完成于 2025-01-10)
  - 耗时: 2天
```

### 四大管理命令 🤖

通过这四个命令，AI 成为你的项目经理。

#### 命令 1: `/start` - 快速进入工作状态

**使用场景：**
- 新打开项目窗口
- 休息一段时间后回到项目
- 需要回忆上下文

**执行过程：**
1. 识别项目类型和技术栈
2. 查看最近 3 条 Git 提交
3. 检查代码状态
4. 扫描业务模块
5. 统计 TODO/FIXME 数量

**输出示例：**
```markdown
# 👋 欢迎回到项目

## 📂 项目信息
- **项目名称**: MyProject
- **技术栈**: Python + Django
- **主要功能**: Web API

## 🕐 最近动态
feat(user): 完成用户管理模块 - 2小时前

## 📊 当前状态
- 代码状态: ✅ 无未提交修改
- TODO: 3个待处理
- 文档状态: 已同步

## 🎯 你可以：
1. 输入 `/progress` - 查看详细进度
2. 输入 `/next` - 获取任务建议
3. 直接说: "继续开发用户模块"
```

**核心价值:**
✅ 1 分钟快速了解状态
✅ 不需要翻 Git 记录
✅ 立刻知道接下来做什么

#### 命令 2: `/progress` - 查看详细项目进度

**使用场景：**
- 产品经理问进度
- 写周报、月报需要数据
- 自查完成情况

**执行过程：**
1. 扫描项目结构
2. 统计功能模块
3. 检查模块完整性
4. 计算完成度百分比
5. 生成详细报告

**输出示例：**
```markdown
# 📊 项目进度报告

## 🎯 项目概况
- 项目名称: MyProject
- 技术栈: Python + Django
- 当前阶段: 开发中
- **整体进度: 72%**

## 📈 功能模块进度

### 后端模块（按功能域统计）

##### 用户管理域 (100%)
1. ✅ 用户认证 - Auth
   - Model: 已完成
   - Service: 已完成
   - API: 已完成

##### 支付模块 🚧 (70%)
1. ✅ 支付配置 - Config
   - Model: 已完成
   - Service: 已完成
   - API: 进行中 (70%)

## 📋 待完成事项
1. [ ] 完成支付接口的测试
2. [ ] 添加 API 文档

## 🔍 代码质量分析
- TODO/FIXME: 4 个待处理
- 代码规范性: 良好
- 文档同步: 已同步
```

**核心价值:**
✅ 精确的进度数据（不是"大概60%"，而是"精确72%"）
✅ 自动生成报告（1 分钟生成）
✅ 可直接复制给产品经理
✅ 按模块分类清晰

#### 命令 3: `/next` - 获取下一步行动建议

**使用场景：**
- 完成一个任务，不知道接下来做什么
- 想知道哪些任务优先级最高
- 需要 AI 帮你规划

**执行过程：**
1. 查看最近 5 条 Git 提交
2. 扫描所有 TODO/FIXME 注释
3. 分析当前开发阶段
4. 按优先级规则排序
5. 给出具体建议

**输出示例：**
```markdown
# 🎯 下一步建议

## 📌 当前状态
- 最近在做: 支付模块 API 开发
- 开发阶段: 中期
- 代码状态: 正常

## 🚀 建议优先级

### 🔴 高优先级（必须马上做）
1. **完成支付模块的 API 测试**
   - 原因: 接口已实现，需要验证功能
   - 预计耗时: 30分钟
   - 位置: `tests/test_payment.py`

### 🟡 中优先级（近期要做）
1. **添加 API 文档**
   - 建议时间: 本周内
   - 位置: `docs/api.md`

### 🟢 低优先级（可以考虑）
1. **优化查询性能**
   - 说明: 提升响应速度，不急
   - 建议时间: 本月内

## 💡 具体行动建议

**建议接下来:**
1. 运行支付模块的单元测试
2. 修复发现的问题
3. 提交代码后运行 `/update-status`

## 📝 注意事项
- 测试时要覆盖边界情况
- 确保环境变量配置正确
```

**核心价值:**
✅ 智能任务调度（AI 帮你排优先级）
✅ 具体可执行（不是"继续开发"，而是"完成测试"）
✅ 扫描 TODO 注释（不遗漏任何任务）

#### 命令 4: `/update-status` - 自动更新项目状态 ⭐⭐⭐⭐⭐

**最强大的功能！开发完一个功能，一条命令自动更新所有文档**

**使用场景：**
- 开发完一个功能模块
- 完成一个重要任务
- 想更新项目进度

**执行过程：**
1. 读取现有项目状态文档
2. 查看最近 10 条 Git 提交
3. 扫描所有业务模块
4. 统计每个模块完成度
5. 扫描 TODO/FIXME 注释
6. 智能更新文档:
   - 将新完成的任务移到"已完成"
   - 更新"进行中"任务的进度
   - 添加新发现的待办任务
   - 更新整体进度百分比
   - 更新时间戳

**输出示例:**
```markdown
# 📝 项目状态更新报告

## 本次更新内容
- ✅ 新增已完成任务：1 项
- 🚧 更新进行中任务：2 项
- 📋 新增待办任务：1 项

## 主要变更

### 1. 支付模块：从进行中 → 已完成
   - 完成时间: 2025-01-14 15:30
   - 完成内容: 接口 + 测试
   - Git: `feat(payment): 完成支付模块`

### 2. 前端页面开发：新增进行中任务
   - 当前进度: 30%
   - 预计完成时间: 2025-01-16

## 📊 进度变化
- 整体进度: 65% → **72%** ↑ (+7%)

---

✅ 文档已更新: docs/项目状态.md
```

**核心价值:**
✅ 再也不用手动维护文档（AI 自动更新）
✅ 精确的进度数据（基于代码自动统计）
✅ 完整的变更记录（每次更新都有报告）
✅ 智能识别新任务（扫描 TODO 自动添加）
✅ 保留手动内容（不会覆盖用户手动信息）

### 进阶技巧

#### 技巧 1: 自定义优先级规则

在 `.claude/commands/next.md` 中定义规则：

```markdown
## 优先级判断规则

### 🔴 高优先级（必须马上做）
1. 有 FIXME 注释的代码
2. 阻塞其他任务的功能
3. 生产环境 Bug
4. 用户反馈的紧急问题

### 🟡 中优先级（近期要做）
1. 已开始但未完成的功能
2. TODO 注释的代码
3. 计划内的新功能
4. 性能优化

### 🟢 低优先级（可以考虑）
1. 代码重构
2. 文档完善
3. 非核心功能优化
```

#### 技巧 2: 团队协作最佳实践

**场景: 多人协作（3-5人团队）**

```
团队成员 A: 开发完功能 → /update-status → git push
团队成员 B: git pull → /progress → 查看整体进度
团队成员 C: /next → 获取任务建议 → 避免冲突
项目经理: /progress → 生成汇报材料
```

**场景: 项目交接**

**老员工交接:**
1. /progress → 生成完整报告
2. 口头补充项目难点
3. git push 所有文档

**新员工接手:**
1. /start → 快速了解项目
2. Read docs/需求文档.md
3. Read docs/项目状态.md
4. /next → 获取可以开始的任务

**交接效率: 节省 80% 时间**

#### 技巧 3: 多项目管理

**场景: 同时负责 3 个项目**

```bash
# 项目切换流程
cd project-a && /start  # 1分钟进入项目A
cd project-b && /start  # 1分钟进入项目B

# 统一进度汇总（每周一）
cd project-a && echo "=== 项目A ===" && /progress

cd project-c && echo "=== 项目C ===" && /progress
```

**价值: 快速切换，不丢上下文**

### 常见问题

**Q1: AI 如何统计项目进度？**

A: AI 自动扫描代码，检查模块完整性:
- Model/Entity → 20%
- Service → 40%
- DAO/Repository → 60%
- API/Controller → 80%
- 测试/文档 → 100%

**示例:**
```
支付模块:
- Model: ✅
- Service: ✅
- DAO: ✅
- API: ❌ (缺少)
完成度 = 60%
```

**Q2: /update-status 会覆盖我手动添加的内容吗？**

A: **不会。** AI 采用智能更新策略：

✅ **保留的内容**
- 用户手动添加的任务
- 自定义的备注和说明
- 手动调整的优先级

✅ **自动更新的内容**
- 从代码分析出的任务状态
- Git 提交记录
- 模块完成度百分比
- 时间戳

**Q3: 如何定义优先级规则？**

A: 在 `.claude/commands/next.md` 中定义规则，AI 根据规则排序任务。

**Q4: 团队成员如何协作？**

A: 推荐两种方式：

**方式1: 共享文档（推荐）**
- 所有文档提交到 Git
- 每个成员开发完运行 `/update-status`
- git push 后其他人 git pull
- 项目经理运行 `/progress`

**方式2: 独立分支**
- 每个成员维护独立分支
- 定期合并到主分支
- 项目经理在主分支查看整体进度

**Q5: 支持哪些项目类型？**

A: 支持所有类型项目，只要：

✅ 必要条件：
- 使用 Git 版本控制
- 有清晰的模块划分

✅ 已测试的项目类型：
- Java Spring Boot
- Python Flask/Django
- Node.js 后端
- Vue/React 前端
- Uniapp 移动端
