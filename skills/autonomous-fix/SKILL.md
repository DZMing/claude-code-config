# 自主修复循环 (Autonomous Fix)

## 核心流程

```
测试失败 → 诊断(读报错) → 定位(3个问题) → 修复(小步快跑) → 验证(全套测试)
```

## 3次尝试机制

- **第1次**：基于分析直接修复（成功率~50%）
- **第2次**：换思路修复（累计~83%）
- **第3次**：回滚并通知用户（累计~93%）

## 诊断步骤

1. **读报错**：从最后一行往上读（最后一行通常是根因）
2. **翻译成人话**：英文报错必须中文解释
3. **记录现象**：
   - 报错现象：[具体描述]
   - 报错位置：[文件名:行号]
   - 报错原文：[英文]
   - 人话翻译：[中文]

## 定位三问

1. **代码逻辑错？** — 变量拼错？条件反了？少 return？运算符错？
2. **环境配置错？** — 依赖没装？端口被占？环境变量没配？
3. **数据问题？** — API 返回空？类型不匹配？边界未处理？

## 修复策略速查

| 错误类型 | 优先策略 | 成功率 |
|---------|---------|-------|
| 语法错误 SyntaxError | 检查括号/引号/逗号匹配 | ~80% |
| 引用错误 ReferenceError | 检查拼写/作用域/导入 | ~70% |
| 类型错误 TypeError | 添加空值检查 `?.` 或 `if (!obj)` | ~60% |
| 断言错误 AssertionError | 修正实现逻辑（不改测试） | ~50% |
| API错误 ECONNREFUSED | 检查服务启动/端口/网络 | ~40% |
| 架构问题 | 停下来重新设计 | ~20% |

## 每次尝试铁律

1. **备份**：修改前 `git add . && git commit -m "chore: 修复前备份"`
2. **一次只改一处**：不要一口气改十个地方
3. **改完立测**：不要攒修改
4. **记录**：每次尝试的方案和结果写入 claude-progress.txt

## 第1次尝试

- 基于错误信息直接定位
- 应用最可能的修复方案
- 立即测试验证

## 第2次尝试

- 回滚第1次修改
- 从不同角度分析问题
- 尝试替代方案

## 第3次尝试（失败则通知用户）

回滚所有修改，汇报格式：

```
老板，自动修复失败，需要介入：
- 现象：[描述]
- 位置：[文件:行号]
- 尝试：
  - 第1次：[方案] → 失败
  - 第2次：[方案] → 失败
- 可能原因：
  - 业务逻辑复杂，需确认需求
  - 架构设计问题，需重新设计
  - 外部依赖问题，需检查网络/API
- 建议：[人工介入方向]
```

## 验证清单

1. 运行原来报错的操作 → 不再报错
2. 运行相关功能 → 没引入新 Bug
3. 运行全套测试 → `npm test` 或 `pytest` 全绿

## 沉淀模板

修好后记录到 claude-progress.txt：

```
> **自动修复成功**：
> **[错误名称]**
> - 现象：[描述]
> - 原因：[根因]
> - 解决：[方案]
> - 教训：[经验]
> - 尝试次数：第N次成功
```

## 成功率预测

| 错误类型 | 第1次 | 累计第2次 | 累计第3次 | 需人工 |
|---------|-------|----------|----------|--------|
| 语法错误 | 80% | 95% | 98% | 2% |
| 引用错误 | 70% | 90% | 95% | 5% |
| 类型错误 | 60% | 85% | 93% | 7% |
| 断言错误 | 50% | 80% | 90% | 10% |
| API错误 | 40% | 70% | 85% | 15% |
| 架构问题 | 20% | 50% | 70% | 30% |

**整体平均**：第1次 ~53%，第2次累计 ~83%，第3次累计 ~93%，需人工 ~7%
